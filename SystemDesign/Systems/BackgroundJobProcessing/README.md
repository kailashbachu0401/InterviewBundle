# 🏗️ Background Job Processing System

## 1️⃣ Problem We’re Solving

We want a system that:

* Accepts long-running jobs
* Does not block clients
* Survives failures, retries, crashes
* Avoids duplicate work
* Is scalable and restart-safe

Examples:

* generate reports
* process files
* send emails
* background computation

---

## 2️⃣ Core Principle (VERY IMPORTANT)

> Request acceptance must be decoupled from request processing.

Clients should not wait for job execution.

---

## 3️⃣ High-Level Architecture
```
Client
  ↓
API Service (stateless)
  ↓
Database (job state + idempotency)
  ↓
Queue
  ↓
Worker(s) (stateless)
  ↓
Database / Object Storage (results)
```

---

## 4️⃣ Key Components & Responsibilities

### 🟦 API Service

* Stateless
* Accepts job requests
* Validates input
* Persists job metadata
* Enqueues job
* Returns job ID immediately

### 🟩 Database (Source of Truth)

Stores all durable state:

* job metadata
* job status
* idempotency keys
* errors / retries
* timestamps

### 🟨 Queue

* Buffers work
* Decouples API from workers
* Provides at-least-once delivery
* Supports retries & visibility timeout

### 🟥 Worker(s)

* Stateless
* Consume jobs from queue
* Execute business logic
* Update job state
* Must be idempotent

---

## 5️⃣ Job Lifecycle (State Machine)

Job States:

* PENDING → created but not yet processed
* RUNNING → worker currently processing
* DONE → completed successfully
* FAILED → permanently failed

State Transitions:
```
PENDING → RUNNING → DONE
              ↓
           FAILED
```

---

## 6️⃣ Job Table (Minimal Schema)
```
jobs
job_id (PK)
status (PENDING / RUNNING / DONE / FAILED)
payload
attempt_count
lease_expires_at
last_error
created_at
updated_at
```

---

## 7️⃣ API Job Submission Flow (SAFE ORDER)

> Persist first, enqueue second

* Validate request
* Create job row (PENDING)
* Enqueue job_id
* Return response with job_id

**Why?**

If API crashes after DB write → job is not lost

Queue publishing can be retried later

---

## 8️⃣ Dual-Write Problem (DB + Queue)

**Problem:**

DB write succeeds

Queue publish fails
→ Job exists but is never processed

**Solutions:**

*Option A: Outbox Pattern*

* Write job + outbox record in same DB transaction
* Background publisher sends outbox events to queue
* Marks outbox row as published

✅ Strong correctness
❌ Extra table + component

*Option B: Simpler Sweeper (Valid Alternative)*

* Add enqueued_at or enqueue_state column to jobs table
* API tries to enqueue and updates state on success
* Background sweeper periodically enqueues jobs still not enqueued

✅ Simpler
⚠️ Slight recovery delay

---

## 9️⃣ Queue Semantics (CRITICAL)

* At-Least-Once Delivery
* Same message may be delivered multiple times
* Duplicates are normal
* System must tolerate them

---

## 10 Visibility Timeout (Queue Concept)

When a worker receives a message:

* Message becomes invisible for X seconds
* If worker ACKs → message removed
* If worker crashes / times out → message becomes visible again

Purpose:

> Prevent multiple workers from processing same job concurrently.

---

## 11 Worker Processing Logic (Correct & Safe)

When worker receives job_id:
```
1 Load job from DB
2 If status = DONE → ACK and exit
3 If attempt_count saturated → ACK and exit
3 Try to atomically acquire lease
  - update status to RUNNING
  - set lease_expires_at = now + lease_duration
  - inc attempt_count by 1
4 If lease acquisition fails:
  - another worker owns job
  - Do NOT ACK
  - allow message to reappear later
5 Execute job logic
```

On success:

* mark DONE
* store result
* ACK

On failure:

* if transient → do NOT mark FAILED, let retry happen
* if permanent → mark FAILED, ACK

---

## 12 Lease / Ownership Model

> DB decides who owns the job, not the queue.

Lease rules:

* RUNNING is temporary
* If lease expires → job can be taken over
* Prevents stuck jobs

---

## 13 Dead Letter Queue (DLQ)

After N failed retries:

* Message sent to DLQ
* Job marked FAILED
* Error recorded

Purpose:

* Avoid infinite retries
* Allow manual inspection

---

## 14 Idempotency Keys (VERY IMPORTANT)

**Problem They Solve**

Client retries same request due to:

* timeout
* network failure
* API crash

Without protection → duplicate jobs created

What is an Idempotency Key?

>A unique identifier for a single logical client intent.

* Generated by client
* Reused on retries
* Sent with request

Example:
```
Idempotency-Key: abc123
```

**API Idempotency Flow**

* API receives request with idempotency key in Header
* Look up key in DB
* If exists:
  - return stored response
* If not exists:
  - create job
  - store idempotency record
* enqueue job
* return response

**Idempotency Table (Minimal)**
```
key (PK)
job_id
response_payload
created_at
expires_at
```

Keys are stored with TTL (e.g., 24h).

---

## 15 Where Idempotency Lives

**API layer:** prevents duplicate job creation

**Worker layer:** prevents duplicate side effects

Idempotency is a system property, not a single function.

---

## 16 Retry Rules (Production Rule)

> Retries are safe only if operations are idempotent.

* Transient failure → retry
* Permanent failure → mark FAILED and stop
* Never blindly retry non-idempotent operations

---

## 17 Why This System is Safe

* Stateless services → easy scaling & restart
* DB as source of truth → durability
* Queue → decoupling & buffering
* Idempotency → correctness under retries
* Leases → no double processing
* DLQ → failure containment
